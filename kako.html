Uvod

Zašto ovo uopće koristimo

Nas šest radi na istom kodu. Bez dogovora — netko će pregaziti tuđi rad, nešto će se slučajno obrisati, i nitko neće znati tko je što napravio i kada.

Tri stvari rješavaju taj problem: Issues (što treba napraviti), Branchevi (izoliran prostor dok radiš) i Pull Requesti (provjera prije spajanja).

Branchevi

Struktura grana

Imamo tri razine:

main ← uvijek radi, igriva verzija, ne dira se │ └── dev ← ovdje se spajaju gotovi featurei │ ├── feature/player-movement ← Pavel radi ovdje ├── feature/level-manager ← Luka radi ovdje ├── feature/boss-battle ← Pavel + Luka ├── art/player-sprite ← Bartol └── sound/sfx-shooting ← Ivan

Nitko ne pusha direktno na main. Programeri ne pushaju direktno na dev — rade na feature branchu pa otvaraju PR. Bartol i Ivan mogu pushati direktno na dev jer PNG i MP3 ne mogu breakati kod.

Tijek:

feature/xxx
→ PR →
dev
→ PR →
main

dev → main ide samo kad više featurea radi zajedno i build je stabilan. Otprilike jednom tjedno.

Svaki dan

Konkretni koraci od nule do merge

01

Uzmi zadnji dev

Uvijek počni s najnovijom verzijom da ne radiš na starom kodu.

git checkout dev
git pull origin dev
02

Napravi svoju granu

Za svaki issue — posebna grana. Ne radi dva featurea na jednoj grani.

git checkout -b feature/player-movement
03

Radi i commitaj često

Svaki commit = jedna logična promjena. Commit poruka počinje glagolom, referencira issue brojem.

git add .
git commit -m "feat: WASD movement in world space (#7)"
git commit -m "fix: normalize diagonal speed (#7)"
04

Pushaj na GitHub

git push origin feature/player-movement
05

Otvori Pull Request na GitHubu

Na GitHubu — Compare & pull request → base: dev. U opisu napiši closes #7 i GitHub automatski zatvori issue kad se mergea. Assignaj jednu osobu za review.

06

Review i merge

Jedna osoba pogleda kod, ostavi komentar ako nešto ne valja ili odobri. Onda merge. Grana se briše, issue je zatvoren.

Pull Request

Kako otvoriti i mergeat PR

PR otvara osoba koja je završila feature. Odobrava netko drugi iz tima — bilo tko od programera, nema posebne hijerarhije, samo ne odobrava sam sebi.

01

Pushaš granu na GitHub

git push origin feature/player-movement

GitHub ti odmah pokaže žuti banner na vrhu repoa: "feature/player-movement had recent pushes" s gumbom Compare & pull request. Klikni to.

02

Popuni formu

Otvori se stranica s nekoliko polja:

  • base: postavi na dev (ne main)
  • Title: npr. Player movement (#7)
  • Description: napiši closes #7 samo ako je taj issue potpuno gotov i više se neće ništa mijenjati na tom dijelu — GitHub će ga automatski zatvoriti pri mergeu. Ako PR rješava issue samo djelomično, nemoj pisati closes, samo referencu: #7
  • Reviewers (desna strana): assignaj jednu osobu, npr. Davida

Klikni Create pull request.

03

Review (David/netko drugi)

Reviewer dobije email/notifikaciju. Ode na GitHub → Pull requests → odabere tvoj PR. Vidi diff — što si promijenio liniju po liniju. Može ostaviti komentar na konkretnoj liniji ili kliknuti Approve.

04

Merge

Nakon approvea, bilo autor ili reviewer klikne Merge pull request → Confirm merge. Grana se briše, issue #7 je zatvoren, kod je u dev.

Ostali u timu trebaju napraviti git pull origin dev da dobiju promjene.

Merge Conflicti

Što je merge conflict i zašto se dogodi

Workflow s branchevima ne garantira da nema conflicata. Conflict nastaje u dva slučaja:

  • Različite grane: Pavel i David oba mijenjaju isti dio game.py na različitim branchevima. David mergea prvi. Kad Pavel otvori PR, Git ne zna čiju verziju uzeti.
  • Ista grana: Dva člana tima rade na istoj grani (npr. oba pushaju direktno na dev). Jedan pusha, drugi ima stariju verziju i kad on pusha — conflict. Ovo je još jedan razlog zašto programeri ne pushaju direktno na dev.
<<<<<<< HEAD (tvoja grana)
    self.enemies = [e for e in self.enemies if e.alive]
=======
    self.enemies = [e for e in self.enemies if e.alive and e.visible]
>>>>>>> dev

Gornji dio (crveno) = tvoj kod. Donji dio (zeleno) = kod iz deva. Ti odlučuješ što ostaje.

Kako riješiti conflict u VS Code

01

VS Code automatski detektira conflict

Kad pokušaš mergeat ili rebaseat i dođe do conflicta, VS Code pokaže fajlove s konfliktom u Source Control panelu (lijeva traka, ikona razgranate ceste). Fajlovi su označeni narančastim ! znakom.

02

Otvori conflicted fajl

VS Code prikaže inline conflict editor s tri gumba iznad svakog konflikta:

  • Accept Current Change — uzmi tvoju verziju
  • Accept Incoming Change — uzmi verziju iz deva
  • Accept Both Changes — zadrži oboje (jedno ispod drugog)

Ako niti jedan gumb ne odgovara — ručno editaš taj dio fajla, obrišeš conflict markere (<<<<<<<, =======, >>>>>>>) i napišeš točno što treba stajati.

03

Commitaj resolution

Nakon što riješiš sve konflikte u svim fajlovima:

git add .
git commit -m "fix: resolve merge conflict in game.py"

Onda nastavljaš normalno — pushaš, PR se ažurira.

Komunicirajte tko radi na kojem fajlu. Ako Pavel i David oba trebaju mijenjati game.py — neka jedan završi i mergea, pa drugi počne od tog. Brancheve ažurirajte često: git rebase origin/dev povuče tuđe promjene u tvoju granu rano, dok su conflicti mali.

Cheatsheet

Brzi referentni popis

# počni novi zadatak
git checkout dev
git pull origin dev
git checkout -b feature/moj-zadatak

# radi, commitaj
git add .
git commit -m "feat: opis (#broj)"

# pushaj
git push origin feature/moj-zadatak

# ažuriraj svoju granu ako je dev napredovao
git fetch origin
git rebase origin/dev

# provjeri gdje si
git status
git log --oneline -10
